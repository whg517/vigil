## 1. 背景与定位

### 1.1 背景

监控/告警系统解决了"**发现问题**"，把指标越界转成一条告警。但告警产生到问题被**真正解决**之间，还有一大段没人管的地带：谁来响应、怎么通知、多人怎么协同、查什么、按什么步骤处置、谁升级、怎么复盘。这段"告警之后"的链条，就是 Vigil 要解决的**告警的"下一步"问题**（详见 [`competitive-analysis.md`](./competitive-analysis.md)）。

### 1.2 产品愿景

> **让每一条告警都被妥善接力到终点。**

### 1.3 产品定位

**开源、IM 原生、AI 原生的告警处置平台**，聚焦告警的"下一步"问题。三个支柱——自托管部署（数据不出企业网络）、本土 IM（钉钉/飞书/企微）深度协同、LLM 横向贯穿分诊/诊断/复盘——同时满足，市场上无同类。

### 1.4 目标

| 维度 | 目标 |
|------|------|
| **用户价值** | 让一线值班工程师被无意义告警打扰的次数减少 50%+，单次事件 MTTR 缩短 |
| **差异化** | 做到"自托管 + 本土 IM 原生 + LLM 原生"三者同时满足，市场上无同类 |
| **开源影响力** | GitHub Star 作为早期传播指标（实现阶段再设硬指标） |

### 1.5 不做（边界）

以下明确不做：

- ❌ **不做监控/采集**：Vigil 只做告警消费者，不内置指标/日志采集——避免与成熟监控方案重复造轮子，集中资源解决"下一步"空白。
- ❌ **不做安全事件响应（SOC/SOAR）**：聚焦运维告警。
- ❌ **不做 SaaS 多租户硬隔离**：单组织、多团队、软隔离。团队是数据归属边界，跨团队协作靠"拉人 + 事件级临时授权"。硬隔离是 SaaS 多租户的需求，本项目不承担其复杂度。
- ❌ **不做商业模式相关切分**：不做 Open Core 付费墙、不做版本功能裁剪。功能完整度不受商业考量影响。
- ❌ **不做全球电话网络自建**：电话/SMS 走云厂商 API。
- ❌ **不做优先级的阶段切分**：本 PRD 描述完整能力，不标 MVP/P0/P1，优先级留待实现阶段再定。

## 2. 用户与场景

目标用户详见 [`personas.md`](./personas.md)。核心用户是**一线值班工程师**与 **SRE 负责人**。

### 2.1 核心用户旅程

```
[告警源] Prometheus/Zabbix/Grafana/云监控/自研
    │  webhook / 邮件 / API 推送告警
    ▼
[能力域 1-2 接入与归一化]  接收 → 归一化为内部 Event
    │
    ▼
[能力域 3 分诊降噪]  去重 / 抑制 / 相关性聚合 → Event 聚合为 Incident
    │
    ▼
[能力域 4 路由]  匹配 Service，绑定升级策略与排班
    │
    ▼
[能力域 5-6 排班与升级]  算出"此刻在班人"，定义升级链
    │
    ▼
[能力域 7 通知]  IM 机器人 / 电话 / SMS / 邮件 / Webhook 送达
    │  超时未 ack
    ▼
[能力域 8 IM 协同 ★]  作战室 / 交互卡片 / 拉人 / 状态双向同步
    │
    ▼
[能力域 9 处置执行]  Runbook（诊断只读内置 / 处置写操作外接或人确认）
    │
    ├─ 全程 → [能力域 10 时间线]  自动捕获留痕
    ├─ 全程 → [能力域 11 AI Copilot]  智能分诊/根因/复盘建议
    │
    ▼ 问题解决
[能力域 12 复盘]  自动草稿 → LLM 结构化 → 改进项跟踪
    │
    ▼
[能力域 13-17 治理/集成/报表/部署/可观测性]  横向支撑
```

Vigil 的价值集中在**分诊 → 路由 → 通知 → 升级 → 协同 → 处置 → 复盘**这条链（能力域 3~12），第 1-2 步只做接入不做产生。

## 3. 完整能力域全景

Vigil 共 **15 个业务能力域 + 2 个横切关注点**。各域的功能需求展开见后续章节，本节先给全景与定位。

| # | 能力域 | 解决的问题 | 核心组成 |
|---|--------|----------|---------|
| 1 | 告警接入（Ingestion） | 告警怎么进来 | 多源 webhook、适配器、邮件、API、接入鉴权 |
| 2 | 事件归一化（Normalization） | 异构告警说同一种话 | 统一 Event 模型、严重度归一、字段映射 |
| 3 | 分诊降噪（Triage） | 哪些该理、哪些该忽略 | 去重、抑制规则、相关性聚合、噪音过滤 |
| 4 | 路由（Routing） | 告警派给谁 | Service 匹配、服务拓扑 |
| 5 | 排班（Scheduling） | 此刻谁在班 | 日历排班、轮班、换班、跨时区、缺席 |
| 6 | 升级策略（Escalation） | 没人理怎么办 | 升级链、超时计时、循环通知 |
| 是7 | 通知（Notification） | 怎么送达 | IM/电话/SMS/邮件/Webhook、模板、送达确认 |
| 8 | IM 协同（ChatOps）★ | 怎么一起干活 | 交互卡片、作战室、拉人、斜杠命令、双向同步 |
| 9 | 处置执行（Runbook） | 该干什么/自动干 | 文档式 runbook、诊断内置、处置外接 |
| 10 | 事件时间线（Timeline） | 全程留痕 | 自动捕获、人工追加、可视化 |
| 11 | AI 智能（Intelligence） | 让人更聪明 | 智能分诊、根因诊断、智能降噪、Copilot |
| 12 | 复盘（Postmortem） | 怎么不再犯 | 自动草稿、LLM 结构化、改进项跟踪、知识沉淀 |
| 13 | 管理治理（Admin） | 谁能干什么 | 用户/团队、RBAC、审计日志、服务目录 |
| 14 | 集成生态（Integrations） | 和外界打通 | 入/双向/出/开放 API |
| 15 | 数据报表（Analytics） | 做得怎么样 | 告警量、MTTR、团队负载、趋势 |
| H1 | 部署与交付（横切） | 怎么跑起来 | Docker/K8s、配置、升级、备份 |
| H2 | 自身可观测性（横切） | 自己健康吗 | metrics/health/日志、吃自己狗粮 |

> 全景图：能力域 1-2 是"入"，3-7 是"分发"，8-9 是"协同处置"，10-11 是"全程贯穿"，12 是"闭环"，13-15+H 是"支撑"。

## 4. 功能需求详述

> 本章按能力域展开完整功能需求（不分优先级）。各域引用的核心实体定义见 [`data-model.md`](./data-model.md)。
> 每个能力域的详细功能设计，将作为独立文档 `docs/capabilities/` 下沉淀；本 PRD 先给出各域的**需求清单与边界**。

### 4.1 能力域 1：告警接入（Ingestion）

**目标**：让任意告警源都能通过标准方式接入 Vigil。

| 需求 ID | 描述 |
|---------|------|
| M1.1 | 提供通用 **Webhook 接收端点**，接收 JSON 告警 payload |
| M1.2 | 内置主流告警源**适配器**：Prometheus/Alertmanager、Zabbix、Grafana、云厂商监控（阿里云/腾讯云/AWS） |
| M1.3 | 提供邮件接入（SMTP → Event） |
| M1.4 | 提供开放 API（程序化推送，供自研系统接入） |
| M1.5 | 接入**鉴权**：每个 Integration 一个 token，请求须校验 |
| M1.6 | 接入点归属 Service，告警默认路由到该 Service（可被路由规则覆盖） |
| M1.7 | 限流与背压：单个接入点流量过大时不拖垮系统 |
| M1.8 | 接收回执：返回接收状态，支持 source_event_id 幂等 |

### 4.2 能力域 2：事件归一化（Normalization）

**目标**：异构告警统一成同一种内部表达。

| 需求 ID | 描述 |
|---------|------|
| M2.1 | 定义**统一内部 Event 模型**（见 data-model §3.3 Event） |
| M2.2 | 每个适配器负责把源 payload **归一化**为 Event |
| M2.3 | **严重度归一**：各源严重度映射到 critical/warning/info，可配置映射表 |
| M2.4 | **字段映射**：源字段 → Event.labels，用于路由匹配 |
| M2.5 | 保留原始 payload 于 `Event.detail`，不丢信息 |
| M2.6 | 归一化失败兜底：无法解析的 payload 落"未识别告警"，告警接入失败要可见可重试 |

### 4.3 能力域 3：分诊降噪（Triage）

**目标**：把 Event 聚合为值得人介入的 Incident，消除噪音。

| 需求 ID | 描述 |
|---------|------|
| M3.1 | **去重**：相同 dedup_key 的重复 Event 合并（含 firing/resolved 配对） |
| M3.2 | **抑制规则**：可配置"满足条件则抑制"（维护窗口、已知问题、低优先级） |
| M3.3 | **相关性聚合**：按 label（service/env）在时间窗口内聚合多个 Event 成一个 Incident |
| M3.4 | **噪音判定**：`Event.is_noise`，被判定噪音的不进 Incident（仅留痕） |
| M3.5 | **自动创建 Incident**：按 Service 的 `auto_create_incident` 配置决定（data-model §3.2） |
| M3.6 | **Incident 合并**：人工或 AI 建议把多个 Incident 合并成一个（`merged_into`） |
| M3.7 | **resolved 处理**：源 resolved 事件自动把 Incident 推向解决态（可配置是否自动 resolve） |

### 4.4 能力域 4：路由（Routing）

**目标**：告警/Incident 找到归属 Service、团队、升级策略。

| 需求 ID | 描述 |
|---------|------|
| M4.1 | **Service 匹配**：按 Event.labels 匹配 Service.labels（精确 + 通配） |
| M4.2 | **路由规则优先级**：多条规则命中时按优先级取一条 |
| M4.3 | **未命中路由**：进入 `unrouted` 池，供人工分诊（需 `event.view_unrouted` 权限） |
| M4.4 | **服务拓扑**：服务依赖关系（可选），用于影响面分析 |
| M4.5 | 路由命中后绑定 Service 的 escalation_policy 与 schedule |

### 4.5 能力域 5：排班（Scheduling）

**目标**：实时回答"此刻谁在班"。

| 需求 ID | 描述 |
|---------|------|
| M5.1 | **日历式排班**：可视化编辑值班日历 |
| M5.2 | **轮班规则**：primary + secondary 多层，支持 daily/weekly/custom |
| M5.3 | **临时换班**：override 层覆盖某时段（需 `schedule.override`） |
| M5.4 | **跨时区**：Schedule 有独立 timezone，团队跨时区正确计算 |
| M5.5 | **缺席处理**：在班人请假/离职能被 override 覆盖 |
| M5.6 | **排班预览**：算出未来 N 天在班人列表 |
| M5.7 | **排班 API**：外部系统查询在班人（供值班大屏等） |
| M5.8 | Schedule 是蓝图，值班人是引擎实时算（不存储快照，见 data-model §3.2） |

### 4.6 能力域 6：升级策略（Escalation）

**目标**：响应不及时，自动找下一个人。

| 需求 ID | 描述 |
|---------|------|
| M6.1 | **升级链定义**：有序 levels，每层 delay_minutes + targets + channels |
| M6.2 | **目标类型**：schedule（在班人）/ user（指定人）/ team（全员）混合 |
| M6.3 | **重复通知**：当前 level 未 ack 前按规则循环通知 |
| M6.4 | **升级时保留上下文**：Incident 上下文随升级流转 |
| M6.5 | **手动升级**：响应者可主动 escalate 跳到下一 level（需 `incident.escalate`） |
| M6.6 | **升级事件**：产生 TimelineItem + IncidentAction 记录 |

### 4.7 能力域 7：通知（Notification）

**目标**：把告警可靠送达。

| 需求 ID | 描述 |
|---------|------|
| M7.1 | **IM 机器人通知**：钉钉/飞书/企微（群机器人 + 私聊） |
| M7.2 | **电话/SMS**：对接云厂商语音 API（阿里云/腾讯云） |
| M7.3 | **邮件通知** |
| M7.4 | **Webhook 通知**：转发到任意自定义端点 |
| M7.5 | **通知模板**：可定制内容格式，按 severity/team 区分 |
| M7.6 | **送达确认与重试**：失败重试，最终失败要可见告警 |
| M7.7 | **ack 闭环**：IM 按钮/Web 回传 ack 状态，回写 Incident |
| M7.8 | **静默时段**：非 critical 在 quiet_hours 不打扰（NotificationRule） |
| M7.9 | **通知聚合**：短时间内多条告警合并通知，避免轰炸 |

### 4.8 能力域 8：IM 协同（ChatOps）★ 差异化核心

**目标**：响应全流程在 IM 内完成，不切系统。一线工程师的"现场"就是 IM 群，ack/升级/拉人/作战室等核心操作都在 IM 内完成，避免切系统降低响应效率；Web 作为管理配置与全局视图的补充。这是 Vigil 最核心的差异化。

| 需求 ID | 描述 |
|---------|------|
| M8.1 | **交互卡片**：告警进 IM 后带 ack/升级/解决 按钮卡片 |
| M8.2 | **一键作战室**：自动建临时群、拉相关人、置顶事件信息 |
| M8.3 | **拉人协同**：卡片上 @ 某人 = `add_responder` + 事件级临时授权（data-model §5.6） |
| M8.4 | **状态双向同步**：IM 内操作实时回写 Vigil，Web 端同步 |
| M8.5 | **斜杠命令**：`/vigil ack <id>`、`/vigil escalate`、`/vigil resolve`、`/vigil runbook <name>` |
| M8.6 | **IM 账号映射**：通过 `User.im_accounts` 把 IM 操作者映射到 User，走标准 RBAC 鉴权 |
| M8.7 | **权限感知卡片**：卡片按钮按用户权限渲染（无权按钮不展示） |
| M8.8 | **多平台支持**：钉钉、飞书、企微（至少两家，依 PoC 定优先级） |
| M8.9 | **作战室归档**：事件关闭后保留作战室聊天记录关联到复盘 |

> ⚠️ 本域能力依赖本土 IM 平台开放能力（卡片交互、@人 API、命令机器人），存在真实风险，须尽早做技术 PoC 验证可行边界。

### 4.9 能力域 9：处置执行（Runbook）

**目标**：响应时知道"该干什么"，并能在受控下执行。Runbook 分两档——诊断类（只读：查日志/指标/拓扑）由 Vigil 内置执行，门槛低；处置类（写操作：重启/扩容/回滚）Vigil 不直接执行，而是生成执行指令、由人确认（human-in-the-loop）或对接用户已有的自动化平台（Ansible/Jenkins/内部平台）。这样避免 Vigil 变成"能搞垮生产的定时炸弹"，把写操作交给用户既有的受控执行环境，权责清晰。

| 需求 ID | 描述 |
|---------|------|
| M9.1 | **文档式 runbook**：Markdown 处置步骤，关联到 Service/Incident 类型 |
| M9.2 | **触发**：manual / on_incident / on_severity / on_label_match（data-model Runbook.trigger） |
| M9.3 | **自动展示**：通知与作战室自动展示关联 runbook |
| M9.4 | **诊断类执行（只读）**：Vigil 内置执行，查日志/指标/拓扑（`readonly:true`） |
| M9.5 | **处置类执行（写）**：生成执行指令，`require_approval:true` 人确认后执行 |
| M9.6 | **外接执行器**：对接 Ansible/Jenkins/HTTP webhook/内部平台（Runbook.steps[].action.target） |
| M9.7 | **执行结果**：回写时间线，失败按 `on_failure` 处理（continue/abort/escalate） |
| M9.8 | **审计**：所有 runbook 执行落 IncidentAction |

### 4.10 能力域 10：事件时间线（Timeline）

**目标**：自动记录事件全程，为协同与复盘打基础。

| 需求 ID | 描述 |
|---------|------|
| M10.1 | **自动捕获系统动作**：状态变更/升级/ack/resolve 等 |
| M10.2 | **自动捕获人工动作**：拉人/备注/状态变更（via=web/im） |
| M10.3 | **Web 可视化时间线** |
| M10.4 | **手动追加**：响应者添加备注/条目 |
| M10.5 | **IM 消息可选捕获**：作战室关键消息回写时间线 |
| M10.6 | 时间线条目类型见 data-model §3.3 TimelineItem |

### 4.11 能力域 11：AI 智能（Intelligence）

**目标**：AI 作为横向 Copilot 贯穿全流程，每条产出带 evidence + human-in-the-loop。分诊时智能聚合、路由时智能建议、处置时给根因线索、复盘时起草——AI 不该是某个开关，而应是 Vigil 的底层能力，与竞品"AI 作为加钱 SKU"形成对比。高风险动作（自动处置）必须有 human-in-the-loop。

| 需求 ID | 描述 |
|---------|------|
| M11.1 | **智能分诊**：建议 Event 合并/Incident 合并（`dedup_suggestion`） |
| M11.2 | **严重度建议**：基于历史与上下文建议调高/调低（`severity_adjustment`） |
| M11.3 | **根因诊断**：处置阶段给根因线索（`root_cause_hint`），引用日志/指标/变更 |
| M11.4 | **相似事件**：检索历史相似 Incident（`similar_incident`） |
| M11.5 | **智能降噪**：学习模式识别噪音（与 M3.4 配合） |
| M11.6 | **摘要起草**：为 Incident 起草 summary（`draft_summary`） |
| M11.7 | **复盘起草**：为 Postmortem 起草结构化内容（`postmortem_draft`） |
| M11.8 | **human-in-the-loop**：所有 AI 产出以 AIInsight 承载，状态 suggested→accepted/rejected 才生效 |
| M11.9 | **可溯源**：每条 AIInsight 必须带 evidence（引用的 Event/日志/时间线） |
| M11.10 | **模型可插拔**：支持云端 LLM 与本地模型（隐私场景） |

### 4.12 能力域 12：复盘（Postmortem）

**目标**：闭环学习，让复盘不再是 4 小时苦差。

| 需求 ID | 描述 |
|---------|------|

## 5. 非功能需求

| 维度 | 要求 |
|------|------|
| **部署** | 自托管优先；Docker Compose + K8s Helm（H1） |

| **可扩展性** | 适配器/通知通道/执行器/AI 模型均可插件化扩展 |

## 7. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| **本土 IM 平台开放能力受限**（钉钉/飞书/企微卡片交互、@人 API 限制） | IM-first 差异化核心受阻 | **优先做 IM 平台能力 PoC**，验证可行边界，必要时调整能力域 8 范围 |
| **LLM 准确率与成本** | AI 产出质量不稳、API 成本高 | 所有 AI 产出走 human-in-the-loop；LLM 缓存 + 限流；支持本地模型（M11.10） |
| **Runbook 写操作风险** | 处置动作误伤生产 | 写操作强制人确认或对接外部平台，Vigil 不直接碰生产写 |
| **自托管部署复杂度** | 用户流失 | Docker Compose 一键部署为硬指标（H1.1） |
| **完整需求范围大** | 设计/实现周期长 | 本 PRD 先完整定义边界；优先级留待实现阶段再定 |

## 附录 A：术语表

| 术语 | 含义 |
|------|------|
| **Event** | 归一化后的原始告警信号（海量、不可变） |
| **Incident** | 值得人介入的处理单元（由 Event 聚合而来） |
| **On-call** | 按排班在特定时段负责响应告警 |
| **Escalation** | 响应不及时自动找下一级 |
| **Runbook** | 处置手册（文档式或可执行式） |
| **MTTR / MTTA** | Mean Time To Resolve / Acknowledge |
| **Postmortem** | 事件复盘 |
| **ChatOps** | 在 IM 内完成运维操作 |
| **RBAC** | Role-Based Access Control，本项目底层权限模型 |
