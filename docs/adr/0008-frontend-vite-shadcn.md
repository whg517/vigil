# ADR-0008: 前端栈选 React + Vite + shadcn/ui + Tailwind

| 字段 | 内容 |
|------|------|
| **状态** | Accepted |
| **日期** | 2026-07-09 |
| **相关** | [`0002-product-positioning.md`](./0002-product-positioning.md)、[`../architecture.md`](../architecture.md)、`web/` |

## 背景

Vigil 前端不是标准中后台表单堆叠,而是偏工具型、需要大量深度定制交互的应用:IM 卡片预览、排班日历(自研,含分层显示与空班检测)、事件时间线、作战室等核心响应页。选型须兼顾开发体验、定制自由度与产物体积。

## 决策

- 技术栈:**React + TypeScript**,**Vite** 构建,**shadcn/ui + Tailwind CSS** 作为组件与样式方案。
- shadcn 组件以源码形式复制进项目(而非黑盒依赖包),对组件行为与样式可完全掌控。
- 用 Tailwind 原子化工具类承载设计系统,新增组件参考 shadcn 风格;前端统一用 `@/` 路径别名,优先复用 `web/src/components/ui/`。

## 理由

- **Vite**:极快冷启动与 HMR、原生 ESM、配置极简,显著优于传统打包器的开发体验。
- **shadcn/ui**:组件源码进项目意味着可完全掌控,契合 Vigil 大量深度定制的交互需求,不受第三方组件库封装边界束缚。
- **Tailwind**:原子化保证设计系统一致性,且产物体积可控(按用到的类生成)。

## 备选方案

- **Ant Design**:明确否决。体积大;定制需覆盖大量默认样式,与本项目"偏工具型、可定制前端"的定位相悖——AntD 更适合标准中后台表单场景,而非 Vigil 这类高度定制的响应界面。

## 影响 / 权衡

- shadcn 组件源码进仓库,后续升级需手动同步上游,换取的是完全的定制自由度。
- 团队需熟悉 Tailwind 原子化写法;换来设计系统一致性与体积可控。
- 前端类型由后端同一份 OpenAPI spec 派生(见集成契约相关 ADR),保证前后端契约不漂移。
