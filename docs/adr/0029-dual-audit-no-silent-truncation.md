# ADR-0029: 双轨审计 + CSV 导出不静默截断

| 字段 | 内容 |
|------|------|
| **状态** | Accepted |
| **日期** | 2026-07-09 |
| **相关** | [`0027-rbac-permissions-roles.md`](./0027-rbac-permissions-roles.md) |

## 背景

告警处置平台涉及权限变更、集成凭据、生产处置等敏感操作,必须可追溯以满足安全合规。同时,Vigil 主打 IM 原生,需要能统计"多少操作是在 IM 里完成的",以量化 IM 作为主交互面的价值。审计数据量可能很大,导出时若为了限流而悄悄截断,会让导出者误以为"导全了",埋下合规隐患。

## 决策

审计分**双轨**:

- **管理审计**:角色变更、集成 token、配置类改动等,查看需 `admin.audit.view` 权限。
- **操作审计**:承载于 `IncidentAction`,记录处置操作,含 `via` 字段(`web | im | api | automation`),可据此统计各入口的操作占比(尤其"多少操作在 IM 完成")。

查询与导出:

- `GET /audit-logs` 分页查询。
- `GET /audit-logs/export` 导出 CSV(org 级,固定列顺序),单次上限 **50000 行**。
- 达到上限时**绝不静默截断**——记 warn 日志,并在响应置 `X-Vigil-Truncated: true` 头,让调用方明确知道结果不完整。

此外,Scoped API Key 由 org_admin 管理(权限 `admin.apikey.manage`),每个 key 绑定各自的权限范围。

## 理由

- 双轨区分"谁改了配置/权限"(管理)与"谁处置了事件"(操作),便于不同角色按需审计。
- `via` 字段让 IM-first 定位可被数据验证。
- 静默截断会被读作"导全了",是误导性合规风险;显式截断标记把"是否完整"交回给调用方判断。

## 备选方案

- **单一审计流**:不区分管理与操作,查询与授权粒度粗,难以精细控制谁能看什么。
- **导出静默截断**:达上限直接停,不给信号——最省事但最危险,直接否决。
- **导出无上限**:一次拉全表,大数据量下拖垮服务,故设 50000 行硬上限 + 截断标记折中。

## 影响 / 权衡

- 调用方必须检查 `X-Vigil-Truncated` 头,否则仍可能误用不完整数据;需在 API 文档中显著说明。
- 50000 行上限意味着超大范围导出需分段(如按时间窗)多次拉取。
- 双轨审计增加写入路径的记录点,处置与配置操作都要确保落审计,不可遗漏。
