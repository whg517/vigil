# ADR-0034: UI/UX 设计原则(oncall 场景)

## 背景

oncall 场景与普通中后台产品本质不同:用户可能半夜被叫醒、需在 30 秒内决策、信息过载会致命、误操作直接影响生产、且常是多人实时协同。这些约束要求 UI/UX 有一套贯穿始终的设计基线,而非套用标准表单式后台。设计调研来源于 incident.io 的 On-call 设计博客。

## 决策

### 四大设计原则(所有 UI/UX 决策须通过)

**半夜能用 · 一屏决策 · 降噪优先 · 状态可见**——任何界面决策都要对照这四条自检。

### 主题与品牌色

- **亮色默认**(尊重后台使用习惯);事件详情等**核心响应页支持暗色**,并在夜间(**22:00–07:00**)首次访问时弹提示做**强引导**。
- 品牌强调色 **Indigo**(primary `#4f46e5`)——冷静专业、与语义色不冲突、契合"守夜人"意象。

### 语义状态色

- **critical / warning / info / success / neutral** 五语义色,各带亮暗双值,映射 Incident 五态。
- **同时带文字标签**,不只靠颜色区分,满足 **WCAG AA** 与色盲可达性。

### 告警卡片

- 摘要用**超大字号 20–24px**(比正文大),首屏只放摘要 + 操作按钮。incident.io 测试发现大字在压力下更易扫读。

### IM 卡片

- **原地更新为主**(依赖平台卡片更新 API,不重发消息)。
- 平台差异走**降级矩阵**:飞书全功能 / 钉钉部分 / 企微降级为发新消息标注最新状态。
- 按当前用户权限渲染按钮(详见 [ADR-0018](./0018-im-same-rbac-as-web.md))。

### 移动端

- **响应式 Web + PWA**(加桌面、推送);移动通知通过 IM 承载。
- **暂不做原生 App**(ROI 低)。

### 组件与国际化

- 排班日历组件**自研**,不用 react-big-calendar——因为 primary/secondary 分层显示 + 空班检测是本产品特殊需求,通用库难以满足。
- **i18n:中文优先**,采用 **i18next** 框架实现全页双语覆盖。

## 理由

- oncall 的极端使用条件(疲惫、时间压力、高风险)要求界面把"快速理解 + 安全操作"放在首位,四原则将这一目标固化为可自检的准绳。
- 亮色默认尊重日常后台习惯,但核心响应页夜间强引导暗色,兼顾多数场景与深夜实际使用。
- 颜色 + 文字双编码是无障碍红线,避免"仅凭颜色传达状态"这一常见可达性缺陷。
- 大字号摘要基于实测:压力下的扫读效率高于常规字号。
- 自研日历是因通用库无法承载分层与空班检测,定制成本低于改造成本。

## 备选方案

- **全站暗色**:否决——多数后台使用场景在白天,全暗违背用户习惯;只在核心响应页夜间强引导暗色更贴合实际。
- **原生 App**:否决——研发/维护成本高而 ROI 低;响应式 Web + PWA + IM 承载移动通知已覆盖移动需求。
- **react-big-calendar 等通用日历库**:否决——无法原生支持 primary/secondary 分层显示与空班检测,改造成本高于自研。

## 影响 / 权衡

- 正面:界面决策有统一准绳,可达性达标,深夜高压场景可用性显著提升。
- 负面/限制:自研日历、双主题、双语覆盖都抬高前端实现与维护成本;暗色仅覆盖核心响应页,非全站一致,需明确边界避免用户困惑。
- 暂不做原生 App 意味着依赖 IM 承载移动推送,移动端体验上限受 PWA 与 IM 平台能力约束。
