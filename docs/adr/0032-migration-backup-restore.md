# ADR-0032： 备份恢复(不做逆向迁移)

| 字段 | 内容 |
|------|------|
| **状态** | Accepted |
| **日期** | 2026-07-09 |
| **相关** | [`0031-single-binary-compose-helm.md`](./0031-single-binary-compose-helm.md)、[`0005-data-access-ent-atlas.md`](./0005-data-access-ent-atlas.md) |

## 背景

Vigil 以 ent 作 ORM、Atlas 作版本化迁移(ADR-0005)。自托管场景下,升级/迁移一旦失败,必须有可靠、可预期的回滚路径——不能让运维在生产上手工拼凑逆向 SQL,也不能依赖脆弱的半自动 down 脚本。核心矛盾:即使迁移是版本化 SQL,实体结构变更(删列/改类型/加约束)的逆向操作仍然无法安全自动化——因为 down 脚本无法恢复被破坏的原始数据(删列后数据已丢,down 加回列也填不回来)。

## 决策

迁移由 Atlas 版本化迁移承载(ADR-0005),`atlas_schema_revisions` 表幂等追踪已应用版本。**不提供 `migrate down`**。升级或迁移失败时,统一走固定处置序列:

```
stop → scripts/restore.sh → 部署回旧版本 → start
```

**升级前必先备份**是回滚的唯一前提。备份恢复由 `scripts/backup.sh` 与 `scripts/restore.sh` 承载,配合 cron 定时执行,保留最近 7 天。

## 理由

- 即使是版本化 SQL 迁移,实体结构变更的逆向仍无法安全自动化(down 加回被删列,数据回不来);维护一套逆向迁移脚本既脆弱又易与实际结构漂移。
- 与其投入成本维护不可靠的 down 脚本,不如统一收敛到"备份即回滚"这条可靠、可验证、语义明确的路径:任何失败都能恢复到升级前的确定状态。
- 备份本就是自托管运维的必备项,复用它承载回滚,不增加额外机制负担。

## 备选方案

- **维护逆向迁移脚本 / 双向 migration**:每个变更同时写 up 与 down。否决——结构变更的 down 无法恢复被破坏的数据,人工维护的 down 脚本脆弱、易漂移,失败场景下反而不可信,给人"可回滚"的错觉。
- **atlas migrate down**:Atlas 自身支持 down 操作,但同样受限于"数据不可逆"原则,仅在「加列/加索引」等纯结构增量场景安全;无法作为通用回滚方案,故不开放。

## 影响 / 权衡

- 正面:回滚路径单一、可靠、可预期;运维心智模型简单(失败就恢复备份);无需为每次 schema 变更额外编写并验证逆向脚本。
- 负面/限制:回滚粒度是"整库恢复到备份点",而非"精确撤销某次迁移";回滚会丢失备份点之后写入的业务数据,因此**升级前必须先备份**是硬前提,遗漏备份则无回滚可言。
- 保留窗口 7 天意味着超过 7 天的历史状态不可恢复,需另行归档策略承载长期留存需求。
