## 背景

Oncall 领域已有大量监控采集、APM、日志平台,它们擅长「发现问题」,却普遍不解决告警产生之后的接力问题:谁来响应、怎么协同、怎么处置、怎么升级、怎么复盘。国内团队还受本土 IM 生态(钉钉/飞书/企微)绑定,现有海外 oncall 工具的 IM 集成停留在「通知通道」层面。需要先明确产品定位与非目标,后续所有能力域设计据此收敛。

## 决策

Vigil 定位为**开源、IM 原生、AI 原生的告警处置平台**,只解决「告警之后的下一步」——响应/通知/协同/处置/升级/复盘,不解决「发现问题」(监控采集)。愿景:让每一条告警都被妥善接力到终点。

- **目标用户**:一线值班工程师、SRE 负责人;中小技术团队(尤其受本土 IM 生态绑定的国内团队)。
- **三大支柱**:① 自托管(数据不出企业网);② 本土 IM 原生(钉钉/飞书/企微,作为协同工作面而非通知通道,最核心差异化);③ LLM 横向贯穿(分诊/诊断/复盘,是底层能力而非付费墙)。
- **设计基线**:接得住不丢失(先落库再处理);软隔离 + 拉人临时授权;AI 全程 HITL + evidence 可降级;写操作不碰生产;RBAC 全程(IM 非后门);可插拔;自身可观测(吃狗粮)。
- **非功能基线**:1000 events/min 接入、通知 P95<5s(目标);用户价值目标:无意义告警打扰次数 ↓ 50%+、单次事件 MTTR 缩短;中文优先并预留 i18n;Docker Compose 一键部署为硬指标。

## 理由

- 告警「下一步」是现有工具的空白地带,聚焦于此形成差异化。
- 本土 IM 作为协同工作面(而非仅通知)是国内团队最痛的缺口,也是相对海外产品的核心壁垒。
- LLM 作为底层能力横向贯穿,与「AI 是加钱 SKU」的竞品形成对比。

## 备选方案

以下均列为 **Non-Goals**,明确不做:

- **监控采集 / APM / 日志**:定位为告警消费者,不与监控平台竞争。
- **SOC / SOAR 安全响应**:领域不同,不扩张。
- **SaaS 硬多租户**:采用单组织多团队软隔离(见 [ADR-0028](0028-single-org-soft-isolation.md)),不做硬隔离。
- **Open Core 付费墙 / 版本裁剪**:能力不做商业裁剪。
- **自建全球电话网络**:电话/SMS 走云厂商 API。
- **无人值守自动修复**:坚持 human-in-the-loop,写操作不碰生产。

## 影响 / 权衡

- 聚焦「下一步」意味着放弃采集侧价值,必须依赖外部告警源接入,接入解耦与不丢失成为硬约束。
- 主要风险:本土 IM 平台开放能力受限(须尽早 PoC);LLM 准确率与成本;Runbook 写操作误伤生产;自托管复杂度。
- 软隔离而非硬多租户,降低了部署复杂度,但跨团队协作须靠拉人 + 事件级临时授权承载。
