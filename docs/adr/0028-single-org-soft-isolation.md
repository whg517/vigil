# ADR-0028: 单组织多团队软隔离 + 团队树不继承权限

| 字段 | 内容 |
|------|------|
| **状态** | Accepted |
| **日期** | 2026-07-09 |
| **相关** | [`0027-rbac-permissions-roles.md`](./0027-rbac-permissions-roles.md)、[`0020-responder-temp-grant.md`](./0020-responder-temp-grant.md) |

## 背景

Vigil 定位为单组织自托管平台,而非面向多租户 SaaS。组织内部按团队划分服务、排班、事件的归属,但团队之间需要频繁临时协作(拉人处置)。若采用 SaaS 式硬多租户,隔离过重、协作受阻、部署复杂;若完全不隔离,又无法界定数据归属与鉴权边界。需要一种"既有边界又能协作"的中间形态。

## 决策

采用**软隔离**:

- **Team 是数据归属边界**——服务、排班、事件均归属团队;资源归属即鉴权作用域(与 [`0027`](./0027-rbac-permissions-roles.md) 的 scope 判定一致)。这不是硬租户隔离,而是同一组织内的软边界。
- **跨团队协作靠拉人**——通过 `add_responder` 拉人,配合事件级临时授权(见 [`0020`](./0020-responder-temp-grant.md)),在不放宽软隔离的前提下受控开放协作。
- **`parent_team_id` 团队树仅用于组织结构展示,权限不沿树向下继承**——父团队管理员不会因层级关系自动获得子团队处置权;跨团队管理须显式通过 org scope 授权。
- **Role 全局定义、所有团队共享**——团队间差异由 RoleBinding 的 scope 承载,不做团队私有角色。

## 理由

- 单组织自托管场景下,SaaS 硬隔离过重且无必要。
- 软隔离 + 事件级临时授权兼顾了"数据边界清晰"与"跨团队能协作"两个目标。
- 权限隐式沿团队树继承是典型越权风险:层级调整会悄悄扩大授权面,难以审计。显式 org scope 授权让跨团队管理有据可查。

## 备选方案

- **SaaS 硬多租户**:强隔离、独立数据面,对单组织自托管是过度设计,且阻碍跨团队协作。
- **权限沿 `parent_team_id` 继承**:管理省事,但隐式扩权、审计困难,与"不越权"基线冲突。
- **团队私有角色**:每团队自维护角色,增加管理负担;差异用 scope 已足够表达。

## 影响 / 权衡

- 团队树只是展示层,不能依赖它做权限推导;所有跨团队授权都要显式建 org scope 的 RoleBinding。
- 跨团队临时协作强依赖 [`0020`](./0020-responder-temp-grant.md) 的临时授权机制,该机制的正确回收是软隔离不被侵蚀的前提。
- 软隔离不防内部越界读取的物理隔离需求,若未来出现强合规隔离诉求需另立 ADR。
