# ADR-0026: 复盘 AI 起草 + 逐字段人工校对 + 改进项外接

| 字段 | 内容 |
|------|------|
| **状态** | Accepted |
| **日期** | 2026-07-09 |
| **相关** | [ADR-0022](./0022-aiinsight-hitl-evidence.md)、[ADR-0024](./0024-similar-incident-pgvector.md)、`ent/schema/postmortem.go` |

## 背景

事件复盘价值高但耗时(通常 3~4 小时),团队常因负担重而拖延或流于形式。AI 能大幅降低起草负担,但复盘质量事关"从故障中学到什么",不能让 AI 自动生成后直接发布——须有人校对把关。

## 决策

复盘状态机:`draft → in_review → published → archived`。

- **触发**:critical 强制复盘——resolved 后自动创建 `draft` 并停在"待复盘"而非直接 `closed`;warning 可配;info 不强制。
- **草稿来源**:三路合成——① 时间线(事实自动填充)② AI 起草(`postmortem_draft` 填 `summary` / `impact` / `root_cause`)③ Incident 元数据。每个字段标注"AI 起草"来源,evidence 引用时间线条目可溯源。
- **逐字段校对**:每个字段可 `accept` / `edit` / `reject`,保证人工把关质量(HITL,见 ADR-0022)。
- **改进项外接**:`ActionItem`(`owner` / `due` / `status` / `TrackerURL`),复盘发布时可经通用 webhook 工单出口在外部工单系统建改进任务,逾期报表高亮。
- **知识反哺**:`published` 复盘进可检索知识库,反哺相似事件检索(见 ADR-0024)。
- **可见性**:默认团队内可见(无指责文化),critical 可设为全公司可见。
- `generated_by`:`ai` | `human` | `mixed`,记录复盘的产出方式。

## 理由

- AI 起草把复盘从 3~4 小时降到约 30 分钟,显著降低负担、提升复盘完成率。
- 逐字段人工校对保证质量,AI 只降负担不替人拍板,坚持 human-in-the-loop。
- 改进项外接既有工单系统,复盘不止于"写完",而是落到可追踪的行动。
- published 复盘沉淀为知识库,与相似事件检索形成正循环。

## 备选方案

- **AI 全自动生成并发布**:质量无人把关,复盘沦为无人信任的自动文本,与 HITL 基线冲突。
- **纯人工从零写复盘**:耗时,导致复盘被拖延或敷衍,失去从故障学习的价值。
- **改进项只记在复盘内**:脱离团队既有工单流,容易被遗忘、无法追踪逾期。

## 影响 / 权衡

- 逐字段校对引入人工环节,牺牲全自动的即时性,换取复盘质量,这是刻意取舍。
- 强制 critical 复盘会让 resolved 后多一个"待复盘"停留态,流程更长但保证关键事件必被复盘。
- 改进项外接依赖外部工单系统的对接与凭证管理(经通用 webhook 出口对接,不做具体厂商 SDK),复杂度上移到集成层。
