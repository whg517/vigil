## 一、设计哲学

### 1.1 oncall 场景的特殊性

oncall 产品与普通中后台有本质差异，根因在于使用场景：

| 普通后台的假设 | oncall 的现实 | 设计影响 |
|---------------|--------------|---------|
| 用户清醒、坐在电脑前 | 半夜被叫醒、手机响应 | 移动端优先、大字号、暗色可选 |
| 有时间慢慢看 | 30 秒内要决策 | 信息层级极强、操作路径最短、一键 ack |
| 信息越全越好 | 信息过载=致命 | 摘要优先、详情按需展开、降噪 |
| 错误可撤销 | 误操作影响生产 | 危险操作强确认、状态明确 |
| 静态展示 | 多人实时协同 | 实时更新、状态可见、谁在做什么 |

调研中 [incident.io 的 On-call 设计博客](https://incident.io/blog/design-details-on-call)专门论证了这些——他们的设计完全围绕"2am 被叫醒的人"，告警屏默认 pitch-black、超大字号只显示摘要、时间线默认折叠。

### 1.2 四大设计原则

Vigil 的所有 UI/UX 决策须通过以下四原则检验：

> **半夜能用 · 一屏决策 · 降噪优先 · 状态可见**

| 原则 | 含义 | 落地 |
|------|------|------|
| **半夜能用** | 凌晨被叫醒、半睡半醒状态下也能完成关键操作 | 告警卡片大字摘要；暗色模式（夜间推荐）；操作按钮明确 |
| **一屏决策** | 用户不必滚动/翻页就能判断"这事多严重、我该干嘛" | 首屏即关键信息（severity + 摘要 + 该做啥）；操作按钮前置 |
| **降噪优先** | 不让噪音淹没真正重要的信号 | 列表分层级；噪音事件折叠；时间线渐进披露 |
| **状态可见** | 任何时候都知道"现在什么状态、谁在做什么、下一步是什么" | 实时状态色 + 责任人 + 时间线 + 协同状态 |

### 1.3 决策记录

- **默认主题**：亮色默认（尊重通用后台习惯），用户可切换暗色。
- **例外**：告警/事件响应类核心页（事件详情、作战室）**支持暗色且夜间强引导使用**——遵循 oncall 场景客观需求。
- **品牌强调色**：**Indigo（靛蓝）**——冷静、专业、可靠，契合"守夜人"气质，且与语义色（红/绿/琥珀）不冲突。
- **实现基础**：shadcn/ui + Tailwind CSS（已搭建设计 token 见 `web/src/index.css`）。

## 二、信息架构

### 2.1 顶层导航（AppShell）

```
┌─────────────┬──────────────────────────────────────────────┐
│  Vigil      │                                              │
│  ─────      │                                              │
│  📊 仪表盘   │              （主内容区）                      │
│  🔴 事件     │                                              │
│  📅 值班     │                                              │
│  🛠  服务     │                                              │

│  ⚙  设置     │                                              │
└─────────────┴──────────────────────────────────────────────┘
  左侧固定导航（可折叠）         右侧自适应内容区
```

导航项与能力域对应：

| 导航项 | 路由 | 对应能力域 | 说明 |
|--------|------|-----------|------|
| 仪表盘 | `/` | 15 报表 | 全局概览 + 快速入口 |
| 事件 | `/incidents` | 3,4,6,8 | 事件列表（按状态分 tab） |
| 值班 | `/oncall` | 5 排班 | 排班日历 + 当前在班 |
| 服务 | `/services` | 4 路由 | 服务目录与路由配置 |
| Runbook | `/runbooks` | 9 处置 | 处置手册管理 |
| 复盘 | `/postmortems` | 12 复盘 | 复盘列表与编辑 |
| 设置 | `/settings` | 13 管理 | 团队/用户/RBAC/集成 |

### 2.2 信息密度策略

| 区域 | 密度 | 理由 |
|------|:----:|------|
| 告警卡片 / 事件详情摘要 | **低**（大字、留白） | 压力场景，扫读 |
| 事件列表 | **中** | 浏览较多，需概览 |
| 配置页（服务/排班/Runbook） | **中高** | 配置场景，清醒操作 |
| 时间线 | **渐进**（默认折叠详情） | 避免过载，按需展开 |

## 三、设计系统

基于 shadcn/ui + Tailwind v4（`web/src/index.css` 已定义 light/dark token）。本节定义扩展到 Vigil 业务语义的设计 token。

### 3.1 色彩系统

#### 语义色（状态色）★ 核心

色彩即状态，无需读字。参考 [Astro UXDS 状态系统](https://www.astrouxds.com/patterns/status-system/)与 [USWDS state tokens](https://designsystem.digital.gov/design-tokens/color/state-tokens/)。

| 语义 | 含义 | 亮色 | 暗色 | 用途 |
|------|------|------|------|------|
| **critical** | 严重/触发 | `#dc2626`（红） | `#f87171` | severity=critical、触发态事件 |
| **warning** | 警告 | `#d97706`（琥珀） | `#fbbf24` | severity=warning |
| **info** | 信息/提示 | `#2563eb`（蓝） | `#60a5fa` | severity=info、提示 |
| **success** | 已解决 | `#16a34a`（绿） | `#4ade80` | resolved/closed 态 |
| **neutral** | 中性/已确认 | `#6b7280`（灰） | `#9ca3af` | acked 态、次要信息 |

**事件状态色映射**（呼应 Incident 状态机）：

| Incident 状态 | 色彩 | 含义 |
|--------------|------|------|
| `triggered` | critical 红 | 刚触发，待响应 |
| `escalated` | critical 红（带"已升级"标） | 超时升级 |
| `acked` | neutral 蓝 | 已接手，处置中 |
| `resolved` | success 绿 | 已解决 |
| `closed` | 中性灰 | 已闭环 |

#### 品牌色（Indigo）

用于 logo、主按钮、选中态、聚焦环：

```
indigo-50  ~ indigo-950 渐变
primary = #4f46e5 (indigo-600)
primary-foreground = #ffffff
```

### 3.2 排版

| 用途 | 字号 | 字重 | 说明 |
|------|------|------|------|
| 告警卡片摘要（★）| **20-24px** | 600 | 压力场景扫读，**比正文大** |
| 页面标题 H1 | 24px | 600 | |
| 区块标题 H2 | 18px | 600 | |
| 正文 | 14px | 400 | 基准 |
| 辅助文字 | 12px | 400 | 时间戳、标签、次要信息 |
| 事件编号（INC-0042）| 等宽 13px | 500 | 等宽便于识别 |

> 关键差异：**告警卡片摘要用大字号**是调研得来的核心洞察——incident.io 测试发现大字在压力下更易扫读，Vigil 沿用。

### 3.3 间距与圆角

| 元素 | 圆角 | 说明 |
|------|------|------|
| 卡片 | `8px`（rounded-lg）| 参考 incident.io 品牌规范 8-20px，取下限保持克制 |
| 按钮 | `6px`（rounded-md）| |
| 输入框 | `6px` | |
| Badge/Tag | `4px`（rounded-sm）| |
| 间距单位 | 4px 基准（Tailwind 默认）| p-2=8, p-4=16, gap-3=12 |

### 3.4 组件规范（基于 shadcn/ui 扩展）

| 组件 | 来源 | Vigil 扩展 |
|------|------|-----------|
| Button | shadcn | 加 `severity` variant（critical 用红、warning 用琥珀）|
| Badge | shadcn | 语义色映射 severity/status |
| Card | shadcn | 告警卡片用专用 `AlertCard` |
| Table | shadcn | 事件列表用 |
| Calendar | 第三方（react-big-calendar 或自研）| 排班日历，支持 primary/secondary 分层 |
| Timeline | 自研 | 事件时间线（见 §4.3）|
| Tabs | shadcn | 事件列表按状态分 tab |

### 3.5 图标

统一用 [lucide-react](https://lucide.dev/)（已安装），线性风格，1.5px 描边。

| 概念 | 图标 |
|------|------|
| 严重/告警 | `AlertCircle` / `Flame` |
| 值班 | `Shield` |
| 排班 | `Calendar` |
| 时间线 | `GitCommit` |
| 升级 | `ArrowUpCircle` |
| 确认 | `CheckCircle` |
| 服务 | `Server` |

## 四、Web 关键页面线框

> 所有线框图为布局与交互逻辑示意，不含最终视觉。`[ ]` 表示按钮，`▼` 表示下拉，`●` 表示状态点。

**目标**：进入即回答"现在有没有事、谁在班、趋势如何"。

│  │ ● 李四 (用户)        │  │  ▲ +1      │  │  ▼ -18%   │  │ ▼ -20% ││
│  └─────────────────────┘  └────────────┘  └───────────┘  └────────┘│
├──────────────────────────────────────────────────────────────────┤
│  活跃事件（需关注）                                                │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │●🔴 INC-0042  [critical] 支付服务 5xx 错误率 > 5%   2m 前    │ │
│  │       值班:张三 · 已升级 1 次 · [查看]                      │ │
│  ├────────────────────────────────────────────────────────────┤ │
│  │●🟠 INC-0041  [warning]  用户中心 P99 延迟升高     8m 前    │ │
│  │       值班:李四 · 已确认 · [查看]                           │ │
│  └────────────────────────────────────────────────────────────┘ │

- 状态点（●）+ severity 色（🔴🟠）一眼识别。
- 趋势图对比"接入量 vs 通知量"，直观看降噪效果。

**目标**：浏览/筛选/批量处理事件。

```

│  事件                            [搜索] [服务▼] [团队▼] [时间▼]   │

│       支付团队 · 张三 · 2m 前 · 升级1次                          │
│  ☐ ●🟠 INC-0041  用户中心 P99 延迟升高    warning│acked           │
│       用户团队 · 李四 · 8m 前                                     │
│  ☐ ●🟢 INC-0040  订单库慢查询已恢复        warning│resolved        │
│       订单团队 · 王五 · 1h 前                                     │
│  ────────────────────────────────────────────────────────────── │
│  选中 2 项:  [批量确认] [批量解决] [分配给...]                    │
└──────────────────────────────────────────────────────────────────┘
```

设计要点：
- **Tab 按状态分**（带计数），切换即筛选。
- 列表行：状态点 + severity + 编号 + 摘要 + 团队/责任人/时间。
- **批量操作**：多选后底部浮出操作栏。
- 噪音 tab 单列，降噪优先。

### 4.3 事件详情（`/incidents/:id`）★ 核心

**目标**：一屏完成"判断 + 操作 + 协同"。这是最高频的响应页面。

```
┌──────────────────────────────────────────────────────────────────┐
│ ●🔴 INC-0042                          [⬆ 升级] [✓ 确认] [...]    │
│ 支付服务 5xx 错误率 > 5%                                          │
│ critical │ triggered · 2m前 · 支付团队                            │
├───────────────────────┬──────────────────────────────────────────┤
│ 左：摘要与上下文       │ 右：时间线 + 协同                         │
│                       │                                          │
│ ▍关联服务            │ ▍时间线 ▼                                 │
│  payment-api (prod)   │  14:02  系统创建事件                       │
│                       │  14:02  告警接入 (prometheus)             │
│ ▍当前值班            │  14:03  升级 level 1 → 通知张三            │
│  张三 (一线)          │  14:04  ● 张三 加入作战室                 │
│  李四 (二线)          │  14:05  AI 根因线索: 可能是 DB 连接 (78%) │
│                       │         [接受] [拒绝]                      │
│ ▍Runbook ▼           │  [+ 添加备注]                              │
│  支付5xx处置.md       │                                          │
│  [一键执行诊断]       │ ▍作战室 (飞书群)                          │
│                       │  本事件作战室 → [打开飞书]                │
│ ▍相关事件            │  成员: 张三 李四 王五                      │
│  INC-0040 (相似 82%) │                                          │

└───────────────────────┴──────────────────────────────────────────┘
```

设计要点：
- **顶部固定操作区**：severity + 编号 + 摘要 + 主操作按钮（升级/确认/更多）始终可见。
- **左栏**：上下文（服务/值班/Runbook/相关事件），AI 根因线索也在此。
- **右栏**：时间线（渐进披露，默认折叠详情）+ 作战室入口。
- **AI 洞察内联**：不单开面板，直接出现在时间线对应位置（human-in-the-loop 按钮）。

## 五、IM 卡片设计 ★ 差异化核心

> IM 卡片是 Vigil 与竞品最大的差异点——一线工程师的"现场"就是 IM 群。
> 详细交互逻辑见 [`capabilities/05-im-chatops.md`](./capabilities/05-im-chatops.md)，本节聚焦**视觉与交互形态**。

### 5.1 告警通知卡片（最核心）

告警进 IM 时发出的交互卡片。**一屏决策**原则的最直接体现。

```
┌─────────────────────────────────────────────────┐
│ 🔴 critical                       INC-0042       │
│                                                  │
│  支付服务 5xx 错误率 > 5%        ← 大字号摘要    │
│                                                  │
│  服务 payment-api · 环境 prod                    │
│  时间 14:02 · 值班 张三                          │
│  Runbook: [📋 查看处置步骤]                      │
│                                                  │
│  [✓ 确认]   [⬆ 升级]   [✓ 解决]   [📋 详情]     │
└─────────────────────────────────────────────────┘
```

设计要点：
- **severity 色条**（顶部 🔴/🟠/🟢）一眼识别严重度。
- **摘要大字号**（参考 §3.2，20px+）——压力场景扫读。
- **操作按钮前置**，按钮按权限渲染（无权的不显示，见 capabilities §5.1）。
- **Runbook 链接**：让响应者立即知道"该干嘛"。

### 5.2 卡片实时状态演进

Incident 状态变化时，卡片**原地更新**（依赖 IM 平台卡片更新 API），不重发新卡：

```
状态1: triggered（刚发）
┌─────────────────────────────────────────┐
│ 🔴 critical            INC-0042          │
│ 支付服务 5xx 错误率 > 5%                 │
│ [✓确认] [⬆升级] [✓解决] [📋详情]        │
└─────────────────────────────────────────┘
              │ 张三点[确认]
              ▼  卡片原地更新 ↓
状态2: acked（已确认）
┌─────────────────────────────────────────┐
│ ✓ 已确认 by 张三        INC-0042         │  ← 标题变绿
│ 支付服务 5xx 错误率 > 5%                 │
│ 张三 处理中 (14:05)                      │
│ [⬆升级] [✓解决] [📋详情]                 │  ← [确认]消失
└─────────────────────────────────────────┘
```

> ⚠️ 平台能力依赖：钉钉/飞书支持卡片更新，企微可能不支持（降级为发新消息标注最新状态，见 capabilities §10）。

### 5.3 多事件聚合卡片

短时间内多条告警聚合（能力域 7 通知聚合），避免轰炸：

```
┌─────────────────────────────────────────────────┐
│ 🔴 3 条支付服务告警 (近 5 分钟)                  │

│  • DB 连接池耗尽           INC-0043              │
│  • 支付网关超时            INC-0044              │
│                                                  │
│  [✓ 全部确认]   [📋 逐条查看]                    │
└─────────────────────────────────────────────────┘
```

### 5.4 作战室入口卡片

## 六、移动端与值班大屏

### 6.1 移动端响应（PWA / 响应式 Web）

oncall 的真实场景大量发生在手机（被叫醒后第一时间掏手机）。Vigil 采用**响应式 Web + PWA**，暂不原生 App（能力域 8 通过 IM 承载移动通知）。

**告警详情页（移动端）**：

```
┌─────────────────────┐
│ ●🔴 INC-0042  [...] │ ← 顶部精简操作
│ 支付服务 5xx        │
│ 错误率 > 5%         │ ← 摘要大字
│                     │
│ critical · 2m前     │
│ 值班: 张三          │
│                     │
│ [✓ 确认]            │ ← 主操作大按钮
│ [⬆ 升级]            │
│ [✓ 解决]            │
│                     │
│ ▍Runbook ▼          │
│ ▍时间线 ▼           │
└─────────────────────┘
```

移动端设计要点：
- **首屏只放摘要 + 操作按钮**，详情下滚。
- 按钮够大（44pt+ 触摸目标），便于半睡状态点击。
- 暗色模式在夜间强引导（亮屏刺眼）。
- PWA 支持加桌面、推送通知（能力域 7 通知的移动兜底）。

### 6.2 值班大屏（Wall Display）

团队工作区投屏用，回答"全队当前状态"。

```
┌──────────────────────────────────────────────────────────────────┐
│  Vigil · 支付团队值班大屏                          2026-06-20 14:30│
├──────────────────────────────────────────────────────────────────┤
│  当前在班                                                        │
│  ┌─────────┐ ┌─────────┐                                        │
│  │ 张 三    │ │ 李 四    │   ← 头像 + 名字，大字                  │
│  │ 一线    │ │ 二线    │                                       │
│  └─────────┘ └─────────┘                                        │
├──────────────────────────────────────────────────────────────────┤
│  活跃事件                                                        │
│  ●🔴 INC-0042 支付5xx 错误率>5%     triggered   2m              │
│  ●🟠 INC-0041 用户中心延迟升高       acked       8m              │
│  ●🟢 INC-0040 订单慢查询已恢复       resolved    1h              │
├──────────────────────────────────────────────────────────────────┤
│  今日指标    接入 42 · 通知 15 · 降噪率 64% · MTTA 2m            │
└──────────────────────────────────────────────────────────────────┘
```

大屏设计要点：
- **字号极大**（3 米外可读），信息极简（只放状态 + 摘要）。
- 自动刷新（WebSocket 实时）。
- critical 事件可闪烁/高亮提醒。
- 全暗背景（不刺眼、省电、突出彩色状态点）。

## 七、交互模式补充

### 7.1 关键快捷键（Web 端）

响应场景下，键盘操作比鼠标快。事件详情页：

| 快捷键 | 动作 |
|--------|------|
| `A` | 确认（ack）|
| `E` | 升级（escalate）|
| `R` | 解决（resolve）|
| `T` | 聚焦时间线备注输入框 |
| `/` | 全局搜索 |

### 7.2 危险操作确认

- 解决（resolve）critical 事件：二次确认。
- 删除事件：输入事件编号确认（防误删）。
- Runbook 写操作执行：必须二次确认（require_approval）。

### 7.3 实时更新

- WebSocket 推送事件状态变化、时间线新增、协同动作。
- 多人同时操作时显示"X 正在编辑/操作"提示（避免冲突）。

## 八、可访问性与国际化

| 维度 | 要求 |
|------|------|
| 对比度 | WCAG AA（4.5:1），状态色在亮/暗模式均达标 |
| 键盘可达 | 所有操作键盘可完成（快捷键 + Tab）|
| 屏幕阅读 | 语义色同时带文字标签（不只靠颜色传达状态）|
| 国际化 | 中文优先，结构预留 i18n；时间显示用户时区 |
