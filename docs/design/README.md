# 功能设计(Design Docs)

> 本目录承载**功能级设计文档**——"docs-driven"流程中「先写设计再写代码」的落笔处。
> 与其他文档的分工(三层正交,详见 [ADR-0001](../adr/0001-record-architecture-decisions.md)):
>
> | 层 | 回答什么 | 放哪 |
> |----|---------|------|
> | 需求/待办 | 要做什么、还剩什么 | [`backlog.md`](../backlog.md) |
> | **功能设计** | **这个功能具体怎么做** | **本目录(一特性一文件)** |
> | 架构决策 | 其中的关键取舍为什么这样定 | [`adr/`](../adr/)(从设计文档中拆出) |
> | 系统全景 | 稳定后的结构长什么样 | [`architecture.md`](../architecture.md)(增量更新) |

## 流程

1. 中等及以上的特性,先在本目录建 `NNNN-<特性名>.md`(编号顺延),按下方模板写设计。
2. 设计中出现**值得单独记录的取舍**(选 A 弃 B 且影响深远)→ 拆一份 ADR,设计文档引用之。
3. 实现合入 main 后:若改变系统全景,增量更新 `architecture.md`;设计文档标记状态为 `已实现`。
4. 小改动(bugfix、样式、单端点微调)不需要设计文档,直接做。

```markdown
- 关联:backlog 条目 / ADR-XXXX / issue

<一段话说清要解决什么;明确不做什么>

## 用户故事 / 触发场景
<谁、在什么场景、期望什么>

## 设计
### 数据模型增量
<新实体/字段,ent schema 伪码;无则写"无">
### API
<新增/变更端点、权限点>
### 状态机 / 流程
<状态流转或时序,文本图即可>
### 前端
<页面/组件增量>

## 边界与失败处理
<幂等、并发、降级、权限边界>

## 测试要点
<单测/e2e 需锁定的行为>
```
